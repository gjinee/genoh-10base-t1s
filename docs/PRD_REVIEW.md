# PRD.md 리뷰

## 좋은 점

- 계층 구조(2.2)가 PHY부터 Application까지 명확히 정의됨
- Key Expression 체계(5.1)가 자동차 존 아키텍처에 잘 매핑됨
- PLCA worst-case 지연 계산(15.2)이 수치적으로 근거가 있음
- 위험 요소 분석(13절)이 실질적이고 완화 방안도 적절
- 학술 논문 근거(IEEE Access, arXiv)로 기술 선택 타당성을 뒷받침

---

## 개선 제안

### 1. 아키텍처 관련

- **Raw Ethernet 전략 불명확**: 2.2절에서 "초기 TCP/UDP → 최적화 단계에서 Raw Ethernet"이라고 했지만, 마일스톤(12절)에 이 전환 시점이 없음. M3 또는 M6에 Raw Ethernet 전환 단계를 추가하는 것이 좋겠음.
- **zenohd와 마스터 앱의 관계**: zenohd(Rust 바이너리)와 Python 마스터 앱이 별개 프로세스로 동작하는데, 이 둘의 통신 방식(같은 호스트에서 TCP loopback? in-process?)이 명시되지 않았음.

### 2. 기능 요구사항

- **FR-006 디스커버리**: Zenoh의 liveliness token 기반 디스커버리를 활용할 것인지, 별도 heartbeat를 구현할 것인지 명확히 해야 함. Zenoh에는 `z_liveliness_declare_token()` / `z_liveliness_declare_subscriber()`가 있어 노드 온/오프라인 감지가 내장되어 있음.
- **FR-008 시나리오**: `wait_sensor`의 `condition` 표현식(예: `"proximity < 30"`)을 어떤 파서로 처리할지 정의 필요. 간단한 비교만 지원할지, 복잡한 룰 엔진이 필요한지.

### 3. 데이터 모델

- **페이로드 중복**: 센서 데이터에 `node_id`, `zone`, `type`이 포함되어 있지만, 이 정보는 이미 Key Expression(`vehicle/front_left/1/sensor/temperature`)에 들어있음. 10Mbps 공유 버스에서 페이로드를 최소화하려면 값(`value`, `timestamp`)만 보내는 것이 효율적.
- **CBOR 사용 기준**: JSON과 CBOR 중 어떤 상황에서 CBOR을 쓸지 기준이 없음. 슬레이브(MCU)는 CBOR이 유리하고, 디버깅 시 JSON이 유리하므로 Content-Type 기반 전환 정책을 추가하면 좋겠음.

### 4. 비기능 요구사항

- **NFR-001** `< 10ms` 목표치: PLCA worst-case가 ~9.7ms인데 Zenoh 처리 시간을 합산하면 10ms를 초과할 수 있음. `< 15ms` 또는 "PLCA 사이클 1회 이내"로 조정하거나, 노드 수별 목표치를 차등 설정하는 것이 현실적.
- **NFR-003** 수집 주기 최소 10ms: 8노드 worst-case에서 10ms가 PLCA 사이클과 거의 동일하므로, 실질적 최소값은 노드 수에 따라 달라짐. 이를 명시하면 좋겠음.

### 5. 구조/운영

- **PLCA 설정 지속성**: 위험 요소(13절)에서 "재부팅 시 초기화"를 언급했지만, FR-001에서도 "자동 재설정"만 언급. `systemd` 서비스 유닛 파일이 구체적으로 어디에 위치할지 모듈 구조(7절)에 반영이 안 되어 있음.
- **에러 핸들링 정책 부재**: FR-001~FR-008 어디에도 실패 시 동작(retry, fallback, 알림)이 정의되지 않았음. 최소한 네트워크 단절, 노드 응답 없음, PLCA beacon 소실 시의 동작을 정의해야 함.
- **보안**: 10BASE-T1S는 물리적 접근이 필요하지만, Zenoh 세션의 인증/암호화(TLS, user-password) 적용 여부가 언급되지 않았음. 시뮬레이션 단계에서는 불필요할 수 있지만, 향후 확장 고려 시 언급할 가치가 있음.

### 6. 다이어그램 정렬

- 2.1절 토폴로지 다이어그램에서 박스 정렬이 약간 어긋나 있음. 모노스페이스 렌더링 시 깨질 수 있으니 확인 필요.
