## 1. 시스템 보안 컨텍스트

### 1.1 대상 시스템 개요

SAM E70 MCU 기반 슬레이브 노드는 10BASE-T1S 멀티드롭 버스를 통해 RPi 5 마스터(zenohd 라우터)와 Zenoh 프로토콜로 통신하며, 헤드라이트/비상등 액추에이터 제어 및 조향 센서 데이터 퍼블리시를 수행한다.

### 1.2 보안 경계 (Trust Boundary)

```
┌─────────────────────── Trust Boundary ────────────────────────┐
│                                                               │
│  ┌─────────────┐        10BASE-T1S Bus        ┌───────────┐  │
│  │   Master    │◄────────────────────────────►│  Slave    │  │
│  │   (RPi 5)   │     Zenoh (IP/UDP/TCP)       │ (SAM E70) │  │
│  │             │                               │           │  │
│  │ ┌─────────┐ │                               │ ┌───────┐ │  │
│  │ │ zenohd  │ │                               │ │zenoh- │ │  │
│  │ │ router  │ │                               │ │pico   │ │  │
│  │ └─────────┘ │                               │ └───────┘ │  │
│  │ ┌─────────┐ │                               │ ┌───────┐ │  │
│  │ │ Safety  │ │                               │ │E2E/   │ │  │
│  │ │ Manager │ │                               │ │SecOC  │ │  │
│  │ └─────────┘ │                               │ └───────┘ │  │
│  └─────────────┘                               └───────────┘  │
│                                                               │
│         ▲ 물리적 접근 경로 (커넥터, 와이어 탭핑)                    │
│         │                                                     │
└─────────┼─────────────────────────────────────────────────────┘
          │
    ┌─────┴──────┐
    │  Attacker  │  ← 외부 위협 행위자
    │  (물리/네트워크) │
    └────────────┘
```

### 1.3 통신 프로토콜 스택

| 계층 | 프로토콜 | 보안 메커니즘 |
|------|---------|------------|
| 응용 | Zenoh Pub/Sub/Query | ACL (Key Expression 기반) |
| 보안 | E2E Protection + SecOC | CRC-32, HMAC-SHA256, SeqCounter, FreshValue |
| 전송 | TCP/UDP | TLS 1.3 (선택적) |
| 네트워크 | IPv4/IPv6 | — |
| 데이터링크 | Ethernet (IEEE 802.3) | — |
| 물리 | 10BASE-T1S (IEEE 802.3cg) | PLCA (결정론적 접근), 물리적 보안 없음 |

### 2.2 자산 의존성 맵

```
AST-005 (액추에이터 명령) ──► AST-002 (HMAC키)로 인증
                           ──► AST-003 (SeqCounter)로 시퀀스 확인
                           ──► AST-004 (FreshValue)로 리플레이 방지
                           ──► AST-012 (ACL)로 권한 확인
                           ──► AST-008 (Safety FSM)에 의해 실행 제어

AST-006 (센서 데이터)     ──► AST-002 (HMAC키)로 서명
                           ──► AST-003 (SeqCounter) 부착
                           ──► AST-004 (FreshValue) 부착

AST-008 (Safety FSM)      ──► AST-001 (펌웨어) 무결성에 의존
                           ──► AST-007 (Zenoh 세션) 상태에 의존
```

## 3. 위협 시나리오 (STRIDE 분석)

### 3.1 STRIDE 모델 개요

| STRIDE 범주 | 위반 보안 속성 | 설명 |
|------------|--------------|------|
| **S**poofing (스푸핑) | 인증 (Authentication) | 신원을 위장하여 시스템 접근 |
| **T**ampering (변조) | 무결성 (Integrity) | 데이터 또는 코드를 무단 수정 |
| **R**epudiation (부인) | 부인 방지 (Non-repudiation) | 행위를 추적/증명할 수 없음 |
| **I**nformation Disclosure (정보 노출) | 기밀성 (Confidentiality) | 민감 정보가 무단 노출 |
| **D**enial of Service (서비스 거부) | 가용성 (Availability) | 정상 서비스 방해 |
| **E**levation of Privilege (권한 상승) | 인가 (Authorization) | 허용되지 않은 권한 획득 |

### 3.2 위협 시나리오 테이블

| ID | STRIDE | 위협 시나리오 | 대상 자산 | 공격 전제 조건 |
|----|--------|------------|----------|--------------|
| THR-001 | Spoofing | **위조 액추에이터 명령 주입**: 공격자가 마스터를 사칭하여 위조된 헤드라이트 OFF 또는 비상등 제어 명령을 10BASE-T1S 버스에 송신. 슬레이브가 이를 유효한 마스터 명령으로 수신하여 실행 | AST-005, AST-008 | 10BASE-T1S 버스 물리적 접근 (와이어 탭핑 또는 커넥터 삽입) |
| THR-002 | Spoofing | **위조 센서 데이터 주입**: 공격자가 슬레이브를 사칭하여 잘못된 조향 센서 데이터를 마스터에 퍼블리시. ADAS가 잘못된 조향 정보로 오작동 | AST-006 | 10BASE-T1S 버스 물리적 접근, Zenoh Key Expression 지식 |
| THR-003 | Tampering | **통신 데이터 CRC/MAC 변조**: 전송 중인 Zenoh 메시지의 페이로드를 변조하여 CRC 불일치 유도, 또는 정교하게 CRC와 데이터를 동시 변조하여 유효한 것처럼 위장 | AST-005, AST-006 | 10BASE-T1S 버스 물리적 접근, 프로토콜 분석 능력 |
| THR-004 | Tampering | **펌웨어 변조**: JTAG/SWD 인터페이스 또는 OTA 경로를 통해 MCU 펌웨어를 악의적으로 수정. 안전 로직 비활성화 또는 백도어 삽입 | AST-001, AST-008 | MCU 물리적 접근 (JTAG), 또는 OTA 서명 키 탈취 |
| THR-005 | Tampering | **E2E 카운터 강제 리셋**: 공격자가 의도적으로 시퀀스 카운터를 리셋하는 메시지를 주입하여 E2E Protection의 연속성 검증을 무력화 | AST-003, AST-004 | Zenoh 프로토콜 지식, 버스 접근 |
| THR-006 | Repudiation | **감사 추적 부재 악용**: 보안 이벤트 발생 시 로그가 기록되지 않거나 공격자가 로그를 삭제/변조하여 사후 추적 불가 | AST-010 | MCU 물리적 접근 또는 펌웨어 취약점 |
| THR-007 | Info. Disclosure | **HMAC 비밀키 추출**: 사이드 채널 공격(전력 분석, 타이밍 분석) 또는 JTAG 덤프를 통해 SecOC HMAC 비밀키를 추출. 키 탈취 후 인증 메커니즘 완전 무력화 | AST-002, AST-009 | MCU 물리적 접근, 사이드 채널 분석 장비 |
| THR-008 | Info. Disclosure | **통신 도청 (Eavesdropping)**: 10BASE-T1S 멀티드롭 특성상 버스 상의 모든 트래픽을 수동적으로 도청. 센서 데이터, 제어 명령 패턴, 키 교환 과정 노출 | AST-005, AST-006, AST-007 | 10BASE-T1S 버스 물리적 접근 (수동 탭핑) |
| THR-009 | DoS | **메시지 플러딩 (Message Flooding)**: 10BASE-T1S 버스에 대량의 Zenoh 메시지를 고속 송신하여 정상 통신 방해. PLCA 타임슬롯 점유 또는 슬레이브 수신 버퍼 오버플로우 유발 | AST-005, AST-006, AST-007 | 10BASE-T1S 버스 물리적 접근, PLCA 노드 ID 할당 또는 CSMA/CD 모드 강제 |
| THR-010 | DoS | **PHY 레벨 재밍 (Physical Jamming)**: 10BASE-T1S 버스에 전기적 간섭 신호를 인가하여 전체 버스 통신 마비. PLCA 비콘 파괴 포함 | AST-005, AST-006 | 10BASE-T1S 버스 물리적 접근, 신호 발생 장비 |
| THR-011 | EoP | **안전 상태 머신(Safety FSM) 우회**: 펌웨어 취약점(버퍼 오버플로우, 포맷 스트링 등)을 이용하여 Safety FSM 상태를 강제 변경. SAFE 상태에서 NORMAL로 부정 전이하여 위험한 제어 명령 실행 가능하게 함 | AST-008, AST-001 | 펌웨어 취약점 발견, 악의적 Zenoh 메시지 조작 |
| THR-012 | EoP | **ACL 우회에 의한 미인가 퍼블리시/구독**: Zenoh Key Expression ACL을 우회하여 슬레이브가 구독하지 않아야 할 제어 채널에 접근하거나, 인가되지 않은 키에 퍼블리시 | AST-012, AST-005 | Zenoh 프로토콜 지식, 버스 접근 |

## 4. 공격 경로 분석 (Attack Path Analysis)

### 4.1 공격 경로 상세

#### AP-001: 위조 액추에이터 명령에 의한 헤드라이트 소등

```
[물리적 접근] ──► [10BASE-T1S 커넥터 삽입]
                      │
                      ▼
              [Zenoh 프로토콜 분석]
                      │
                      ▼
            [Key Expression 패턴 파악]
            vehicle/{zone}/{node_id}/actuator/headlight
                      │
                      ▼
        [위조 Zenoh PUT 메시지 생성] ──► SecOC 미적용 시: 즉시 성공
                      │                  SecOC 적용 시: HMAC 검증 실패로 차단
                      ▼
            [슬레이브 수신 및 실행]
                      │
                      ▼
            [헤드라이트 소등 → HAZ-002]
```

- **공격 실현 가능성**: SecOC 미적용 시 **높음**, SecOC 적용 시 **매우 낮음**
- **필요 장비**: 10BASE-T1S 인터페이스 보드 (예: EVB-LAN8670-USB), 노트북
- **필요 시간**: 프로토콜 분석 수일, 공격 실행 수초
- **필요 지식**: Zenoh 프로토콜, 10BASE-T1S PHY 계층

#### AP-002: JTAG을 통한 HMAC 키 추출 및 완전한 인증 무력화

```
[MCU 물리적 접근] ──► [JTAG/SWD 커넥터 연결]
                          │
                          ▼
                    [Flash 메모리 덤프]
                          │
                          ▼
                 [HMAC 키 바이너리 검색]
                          │
                          ▼
                [키 추출 성공] ──► JTAG 비활성화 시: 실패
                          │
                          ▼
            [완전한 메시지 위조 가능]
            (유효한 HMAC 서명 생성)
                          │
                          ▼
          [THR-001 + THR-002 모두 실현 가능]
```

- **공격 실현 가능성**: JTAG 활성화 시 **높음**, JTAG 비활성화 + Flash 보호 시 **낮음**
- **필요 장비**: JTAG 프로브 (J-Link, ST-Link 등), 디버거 소프트웨어
- **필요 시간**: 수시간 (키 위치 파악 포함)
- **필요 지식**: ARM Cortex-M7 아키텍처, Flash 메모리 레이아웃

#### AP-003: 메시지 플러딩에 의한 서비스 거부

```
[10BASE-T1S 버스 접근] ──► [PLCA 노드로 참여 또는 CSMA 모드 강제]
                               │
                               ▼
                  [고속 Zenoh 메시지 연속 송신]
                  (10Mbps 대역폭의 50% 이상 점유)
                               │
                               ▼
                  [정상 노드 통신 지연/실패]
                               │
                               ├──► [센서 데이터 전달 지연 → HAZ-004]
                               └──► [제어 명령 전달 지연 → HAZ-007]
                                        │
                                        ▼
                               [안전 상태 전이 (SAFE)]
```

- **공격 실현 가능성**: **중간** (물리적 접근 필요하나 기술적 난이도 낮음)
- **필요 장비**: 10BASE-T1S 인터페이스 보드, 패킷 생성 스크립트
- **필요 시간**: 수분 (세팅), 공격 지속 시 실시간
- **필요 지식**: 기본적인 네트워크 지식

#### AP-004: 사이드 채널을 통한 암호키 추출

```
[MCU 물리적 접근] ──► [전력 분석 프로브 설치]
                          │
                          ▼
                [HMAC 연산 중 전력 소비 패턴 수집]
                (수만 회 반복 측정)
                          │
                          ▼
                [DPA(Differential Power Analysis)]
                          │
                          ▼
                [HMAC 키 복원] ──► 성공 확률: 전용 장비 시 중간
                          │
                          ▼
                [AP-002 이후 공격 경로와 동일]
```

- **공격 실현 가능성**: **낮음** (고가 장비, 전문 지식, 장시간 필요)
- **필요 장비**: 오실로스코프(고대역), 전력 분석 소프트웨어 (ChipWhisperer 등)
- **필요 시간**: 수주 ~ 수개월
- **필요 지식**: 사이드 채널 분석 전문가 수준

#### AP-005: 안전 상태 머신 우회 공격

```
[Zenoh 메시지 퍼징] ──► [비정상 페이로드 송신]
                           │
                           ▼
                  [zenoh-pico 파서 취약점 트리거]
                  (버퍼 오버플로우, 정수 오버플로우)
                           │
                           ▼
                  [임의 코드 실행 또는 메모리 손상]
                           │
                           ▼
                  [Safety FSM 상태 변수 조작]
                  (SAFE → NORMAL 강제 전이)
                           │
                           ▼
                  [위험 상태에서 액추에이터 제어 가능]
```

- **공격 실현 가능성**: **낮음** (zenoh-pico는 C 구현이므로 메모리 취약점 가능성 존재하나, 입력 검증 구현됨)
- **필요 장비**: 10BASE-T1S 인터페이스, 퍼징 프레임워크
